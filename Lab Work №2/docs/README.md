# Лабораторная работа №2
## Тема: 
Использование нотации C4 model для проектирования архитектуры программной системы
## Цель работы:
Получить опыт использования графической нотации для фиксации архитектурных решений. 
## Диаграмма системного контекста 
![c1](/images/c1.png)

Пользователи (студенты, волонтёры и администраторы) взаимодействуют с Telegram-ботом через клиентское приложение Telegram. Все сообщения сначала поступают в Telegram API, который перенаправляет их в систему. В ответ бот формирует сообщения, которые Telegram API доставляет обратно пользователям. Таким образом, Telegram API является единственным каналом связи между пользователями и системой.

## Диаграмма контейнеров 
![c2](/images/c2.png)

В проекте используется модульная сервисная архитектура, в которой система разделена на несколько независимых контейнеров по функциональным областям. Такой подход позволяет изолировать пользовательский и админский блок, что повышает безопасность и упрощает поддержку системы, а также масштабировать её.

Диаграмма контейнеров показывает внутреннюю структуру системы и распределение ответственности между её основными подсистемами. Система включает два независимых backend-контейнера: пользовательский Telegram-бот, обрабатывающий запросы студентов, и административный бот, предназначенный для работы волонтёров и администраторов. Дополнительные контейнеры обеспечивают генерацию QR-кодов, обработку заданий и голосований, а общая база данных PostgreSQL хранит всю необходимую информацию о пользователях и мероприятиях. Взаимодействие с пользователями осуществляется исключительно через Telegram API.

## Диаграмма компонентов 
![c3.2](/images/c3.png)

Диаграмма компонентов детализирует внутреннее устройство двух основных контейнеров системы: пользовательского и административно-волонтёрского Telegram-ботов. В пользовательском контейнере выделены компоненты, отвечающие за маршрутизацию команд студентов, регистрацию участников, отображение расписания, работу с заданиями и голосованиями, а также получение персонального QR-кода. 

Административный контейнер включает модули авторизации, фиксации посещаемости, а также компоненты управления заданиямии и голосованиями, генерацией кодов доступа. 

Каждый контейнер взаимодействует с Telegram API через собственный компонент-коннектор и обращается к соответствующим сервисам системы для выполнения бизнес-логики. Такое структурирование позволяет чётко разделить обязанности модулей и упростить поддержку и расширение функциональности.
