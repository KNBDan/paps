# Лабораторная работа №1

## Тема: 
Формулирование требований к программной системе

## Цель работы: 
Научиться анализировать поставленную задачу, формулировать функциональные и нефункциональные требования к проектируемой системе.

## Перечень заинтересованных лиц (стейкхолдеров)
- Студенты- участники конференции. Потенциальные посетители проектной ярмарки, мастер-классов и мирового кафе. Используют чат-бот для навигации по мероприятию, получения информации, участия в голосованиях и заданиях, а также показывают волонтёрам свой QR-код для фиксации посещения.
- Волонтёры мероприятия. Помогают организаторам, используют чат-бот для сканирования QR-кодов студентов и фиксации посещаемости.
- Администраторы мероприятия. Отвечают за контроль процесса проведения мероприятия.
Имеют доступ к расширенным функциям бота, такими как: запуск голосований, запуск заданий, генерация кодов для авторизации волонтёров.
- Организаторы конференции. Ответственны за общее проведение мероприятия, подготовку расписания, мастер-классов, мирового кафе и ярмарки проектов. Им нужна статистика посещаемости и вовлечённости участников.
- Представители компаний. Проводят мастер-классы, участвуют в мировом кафе. Им важен поток студентов, статистика посещения их зон.
- Технические специалисты / разработчики. Поддерживают корректную работу чат-бота, баз данных и административных панелей.

## Перечень функциональных требований
Функциональные требования для студентов
1) Регистрация на мероприятие.
2) Просмотр расписания мероприятий, проектов, мастер-классов.
3) Генерация личного QR-кода участника.
4) Получение уведомлений.
5) Участие в голосованиях.
6) Просмотр списка активных заданий.
7) Отправка ответов (текст, фото, видео, в зависимости от задания).
8) Просмотр карты мероприятия (локации, зоны мастер-классов, мировое кафе, ярмарки).


Функциональные требования для волонтёров
1) Авторизация с использованием специального кода (полученного от администратора).
2) Сканирование QR-кода студента через бот.
3) Автоматическая фиксация посещения определённой активности.
4) Возможность выбора активности (мастер-класс, проект, стенд компании), на которой производится отметка.


Функциональные требования для администраторов
1) Авторизация в административном режиме с использованием кода (полученного от разработчиков).
2) Запуск голосований (например: «Лучший проект ярмарки»).
3) Завершение голосований (подведение итогов).
4) Запуск временных заданий.
6) Генерация кодов для авторизации волонтёров.
7) Просмотр списка доступных заданий студентов.
8) Оценка сданных работ студентов.
9) Завершение конференции.

## Перечень сделанных предположений
- Все взаимодействия пользователей с системой происходят только в Telegram, и дополнительных веб-интерфейсов не предусмотрено.
- Волонтёры используют встроенный мини-приложение в Telegram для сканирования QR-кодов студентов.
- Администраторы получают присланные ответы студентов в отдельный чат, где они могут оперативно просматривать ответы.
- Изменения в данных о мероприятиях, мастер-классах, заданиях, компаниях и других сущностях выполняются только разработчиками вручную.
- Результаты конференции собираются исключительно через выгрузку данных из базы разработчиками; в боте нет механизмов автоматической аналитики.
- Предусмотрен инструмент переноса отметки посещения между последовательными мероприятиями, чтобы избежать повторного сканирования студентов при переходе между двумя связанными активностями.
- Администраторы могут запускать временные задания (бириллы), которые автоматически закрываются после истечения установленного времени.

## Перечень нефункциональных требований
- Надёжность.
Бот должен стабильно работать весь день конференции без критических остановок, а кратковременные сбои должны устраняться автоматически; высокая надёжность необходима, чтобы фиксировать посещаемость без потерь данных.

- Масштабируемость.
Система должна корректно работать при количестве студентов до тысячи и десятка волонтёров, что напрямую связано с надёжностью и предотвращает сбои при пиковой нагрузке.

- Удобство использования.
Интерфейс должен быть максимально простым: кнопки, минимальное число действий, понятная навигация, чтобы волонтёры могли быстро отмечать студентов, а участники не испытывали сложностей при взаимодействии с ботом.

- Безопасность.
Авторизация волонтёров должна выполняться через одноразовые или уникальные коды, доступ администраторов-  также, а самые ответственные функции (запуск голосований, заданий, генерация кодов) должны быть доступны исключительно администраторам.

- Производительность.
Бот должен отвечать в течение 1 секунды при нормальной нагрузке и до 3 секунд в пик, чтобы процесс отметки посещаемости и взаимодействия с ботом происходил быстро и не создавал очередей.
